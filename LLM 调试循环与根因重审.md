# LLM 调试循环与根因重审

## 最初的问题

写代码报错，反复修改后还是困在同一个问题里出不来。这是命中了缓存吗？

## 对话展开

### 不是缓存，是推理锚定

小墨说：缓存是结果层面的锁定，这个是**推理层面的锁定**。LLM 不是重复返回缓存内容——每次都在生成新内容——但推理方向被第一轮的诊断锁死了。

### 三个原因叠加

**1. 诊断锚定。** 第一轮给出的判断像锚，后续推理被拖住了。LLM 不会「推翻重来」——逐 token 生成过程，开头思路决定结尾方向。

**2. 上下文自我强化。** 每轮对话积累「试了 A、B、C 都不行」的证据，但全指向同一个方向（比如数组越界）。LLM 不是质疑方向，是继续加码。

**3. 没有外部打断。** 人类卡住五轮会站起来喝杯水，回来突然想「是不是栈溢出」。LLM 没有重启按钮。

### 临时破局

最有效的一招——**强制换方向**：
「之前的思路全不对，跳出当前假设，列出其他所有可能的原因」

这句话的核心是强行重置推理锚点。或者开新对话，只描述报错现象不提试过什么。

### 系统级解法

Fix Loop 设硬上限——同一个错误类型连续 3 次，自动触发根因重审：
1. 不继续修
2. 强制走分支：「放弃当前假设，从错误栈中提取新线索」
3. 列举所有可能根因，按可能性重新排序

## 关键转折

### LLM 不会自我学习

小墨纠正了一个关键误解：训练完成后权重就冻结了。对话不会改变模型。修好只发生在当前上下文窗口里——关掉对话就清零。

| 你以为的 | 实际的 |
|----------|--------|
| 用多了就熟了 | 每次都是第一次见你 |
| 改了就能记住 | 关掉对话就忘了 |
| 越用越聪明 | 聪明的是你，不是模型 |

### Agent 能做什么

Agent 不是让模型学习，是**用外部机制补模型的失忆症**：
- Memory：记住上次踩过的坑
- Skill：把反复命中的经验变成可复用流程
- Session Search：找回完整的纠偏过程

具体经验越丰富，复发概率越低。一次完整纠偏记录 → 复发 30-40%。两次 → 10-20%。

## 现在的理解

缓存困结果，锚定困推理——本质相同：系统没有「我可能错了」的感知开关。解法框架也一样：给系统一个自我怀疑的触发阈值。缓存靠质量系数降权，推理靠强制根因重审。
