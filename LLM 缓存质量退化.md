# LLM 缓存质量退化

## 最初的问题

用户上次的问题回答得不好，他修改了措辞重新问。但语义相似度高，缓存命中了，返回的还是那个烂答案——永远改不了。这是不是缓存系统的致命缺陷？

## 对话展开

### 缓存的核心假设是错的

缓存的假设是「同样的回答是好的」。但用户改措辞重问，意味着**上次的回答没解决问题**。缓存不知道这个信号——它只看相似度。

### 解法一：命中后追问（最轻量）

缓存命中 → 返回旧答案 + 追问「满足吗？」→ 用户点「没有」→ 缓存失效重新生成 → 新答案覆盖。

成本接近零，但需要交互。

### 解法二：质量系数降权（自动化）

```
缓存得分 = 语义相似度 × 质量系数

质量系数：
  新缓存：1.0
  用户点赞：1.0
  用户无反馈：0.8
  用户差评：0.0（淘汰）
  被追问过：0.5

命中阈值 0.9 → 打折后 < 0.9 → 不命中
```

关键洞察：**不需要用户主动反馈，用户追问本身就是负反馈信号。**

### 解法三：追问链检测（最智能）

同一 session 内，连续两个问题语义相似但措辞不同 → 反推上一次没解决。不需要显式反馈，但需要维护 session 内的追问链状态。

### 方案对比

| 方案 | 复杂度 | 用户体验 | 适用 |
|------|--------|---------|------|
| 无反馈 | 最低 | 差 | 不推荐 |
| 命中后追问 | 低 | 好 | 快速迭代首选 |
| 质量系数降权 | 中 | 好 | 有追问数据的场景 |
| 追问链检测 | 高 | 最好 | 成熟产品 |

## 关键转折

理解了缓存应该从「不可变的历史快照」变成「会根据反馈自我纠错的活系统」。这和后面聊的调试根因重审、Agent Memory 降权是同一个设计模式——系统需要一个「我可能错了」的感知开关。

## 现在的理解

缓存质量退化的问题解法不在缓存本身，在**反馈闭环**。有反馈就有降权，有降权缓存就能自救。最简单的落地：追问检测 + 质量系数打折。
