# RAG 查询理解

## 最初的问题

我用口语化的语言描述问题（专业名词少、表达随意），怎么做到精准搜索？是大模型先去进行语义理解，把口语翻译成文档能听懂的话，然后再检索吗？

## 对话展开

### 为什么直接搜不行

小墨举了个例子：

```
用户说：「那个能自己写代码的工具是咋回事」
文档里写：「AI-powered code generation via autonomous agent frameworks」
```

向量相似度极低，但说的就是同一个东西。因为向量模型把「语义相近」理解为「词在训练数据里经常共现」，而口语表达和专业文档表述根本不在同一个语义空间。

### 三种解法

**查询改写（Query Rewriting）**：最常用。检索之前让 LLM 把口语转成搜索关键词：

```
用户：「怎么让模型不瞎编」
  ↓ LLM 改写
搜索查询：「大语言模型 幻觉 hallucination 缓解方法」
```

成本极低（一次小模型调用），效果显著。大部分生产系统至少会做这一步。

**多角度检索（Multi-Query）**：一个查询拆成 3-4 个不同角度的搜索语句，分别检索再合并去重。覆盖面更广。

**HyDE（假设文档嵌入）**：更激进的思路。让 LLM 先「瞎编」一个理想答案，拿这个假答案去搜。理由是：答案和文档段落的语义空间一致，比问题更容易命中。

### 方案对比

| 方法 | 成本 | 效果 | 什么时候用 |
|------|------|------|------------|
| 直接搜 | 零 | 差 | 用户用了精确关键词 |
| 查询改写 | 极低 | 好 | 大部分场景，默认该加 |
| 多角度检索 | 低 | 更好 | 问题复杂、文档库大 |
| HyDE | 中 | 最好 | 专业领域、冷门问题 |

## 现在的理解

在检索之前通常需要 LLM 做一次语义转换，不是把用户的话直接向量化，而是先翻译成「文档能听懂的话」。这一步被称为 Query Understanding 或 Query Transformation，是整个 RAG 链路里投入产出比最高的优化点。
