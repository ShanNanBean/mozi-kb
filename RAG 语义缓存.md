# RAG 语义缓存与答案一致性

## 最初的问题

为什么每次检索返回的答案都不一致？每次都需要重新去查询吗？Obsidian 那种「上次问过的答案下次直接返回」的模式是不是比传统 RAG 更好——更省 token、用户体验更好？

## 对话展开

### 为什么不一致

两个原因：
1. **LLM 本身是随机的**：同样 prompt 问两次，采样策略（temperature）导致措辞不同
2. **检索结果可能漂移**：向量检索是近似搜索，同一个 query 两次可能回传不同的 Top-K

两者叠加：检索波动 × 生成随机 = 答案不稳定。

### 语义缓存的做法

```
第一次：「RAG 怎么提高检索精度」→ 检索 → LLM 生成 → 存缓存
第二次：「RAG 检索怎么更准」→ 语义相似度 0.92 → 命中 → 直接返回
```

省的不只是 token——体验差别是质的改变。问过的问题秒回，用户会本能地信任系统。

### 三种缓存失效策略

| 策略 | 做法 | 优劣 |
|------|------|------|
| TTL 过期 | 设 24 小时自动失效 | 简单但粗糙，新文档可能不生效 |
| 文档指纹感知 | 缓存记录引用的源文件+修改时间，变更则重新检索 | 准确，但检查成本接近重新检索 |
| 增量生成 | 旧答案当底稿，只追加新文档的增量信息 | 省 token，但无法处理新旧矛盾 |

### 更深的问题：不只是缓存

Obsidian 那种模式更进一步——每次问答沉淀成可检索的知识条目，形成「个人知识反馈环」：
- 重复提问秒回
- 答案稳定一致
- 隐式偏好学习（你追问的方向、纠正过的答案，系统记住了）
- 知识压实（LLM 提炼过的答案比原始碎片质量高）

## 关键转折

意识到这不仅是技术优化，是**交互模式的改变**。传统 RAG 的「每次从零检索」其实不自然——人也不是每次都从头查资料，学过的就记住了，下次直接调用。

## 核心权衡

缓存策略选择取决于文档库更新频率。频繁更新 → 缓存意义不大。相对稳定 → 文档指纹策略最优。没有完美方案，只有适合场景的取舍。
