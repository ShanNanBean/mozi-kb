# LLM API 设计：流式、缓存、重试、路由

## 最初的问题

一个 `openai.chat.completions.create()` 调通了不代表能上生产。真正的 LLM 应用要在**稳定性、成本、速度、用户体验**四个互相矛盾的目标之间做架构取舍。我想搞清楚生产级 LLM 调用的完整链路是什么样的。

## 对话展开

### 完整链路

```
用户输入 → 缓存检查 → 路由决策 → 限流排队 → 发送请求 →
重试判断 → 流式回传 → 结果缓存 → 用户看到回答
```

每一步都是一个独立的架构决策。

### 第一层：缓存

不是存 key-value。**语义缓存**——「RAG 怎么优化」和「检索增强生成咋调优」应该命中同一个结果。

| 缓存策略 | 做法 | 命中率 | 成本 |
|----------|------|--------|------|
| 精确匹配 | query 完全一样才命中 | 低 | 零 |
| 语义相似 | embedding 相似度 > 0.9 | 高 | 一次 embedding |
| 多级缓存 | 精确→语义→LLM 三级回落 | 最高 | 阶梯递增 |

关键判断：如果 embedding 成本是 LLM 的 1/1000，语义缓存几乎零成本白嫖。

### 第二层：路由

不是所有问题都需要最强模型。分类路由是省成本的核心：

```
用户输入 → 意图分类（小模型）
  ├── 简单闲聊 → GPT-3.5
  ├── 技术问答 → GPT-4 / Claude
  ├── 代码生成 → Claude Opus / DeepSeek
  └── 敏感内容 → 拒绝
```

| 路由策略 | 逻辑 | 代价 |
|----------|------|------|
| 意图路由 | 分类后分发 | 多一次分类调用 |
| 成本路由 | 优先用便宜模型 | 复杂问题可能答不好 |
| 故障路由 | 主模型超时→切备厂商 | 多厂商 key 维护 |
| 延迟路由 | 优先用最快的 | 质量可能打折 |

### 第三层：重试

核心原则：**不要用同一个模型以同一种方式重试同一个错误。**

| 错误类型 | 策略 | 原因 |
|----------|------|------|
| 429 Rate Limit | 指数退避，最长等 30s | 等厂商放量 |
| 5xx 服务端错误 | 立即切备用模型 | 这个模型大概率短期恢复不了 |
| 4xx 客户端错误 | 不重试 | token 超长、参数错误，重试没用 |
| 超时 | 重试一次→还是超时就切 | 别让用户等超过 10 秒 |
| 内容过滤 | 换个说法重试一次 | 可能是误判敏感词 |

### 第四层：流式 vs 非流式

| | 流式 | 非流式 |
|---|---|---|
| 首字延迟 | <500ms | 3-30s |
| 实现复杂度 | SSE/WebSocket | 简单 |
| JSON 返回 | ❌ 不完整没法 parse | ✅ |
| 中断处理 | 连接断开即停 | 已生成的 token 仍计费 |

**关键决策**：Chat 用流式，API 返回 JSON 用非流式。成熟的方案是**流式生成 + 非流式落盘**——用户看到流式输出，后台等全文生成完才存库。

### 生产标准对比

| | 最小可行 | 生产标准 | 高可用 |
|---|---|---|---|
| 缓存 | 无 | 精确匹配 | 语义缓存+TTL |
| 路由 | 单模型 | 意图路由 | 三层路由 |
| 重试 | 无 | 固定退避 | 差异化+切备 |
| 流式 | 非流式 | SSE | 流式生成+非流式落盘 |

## 关键转折

意识到每一层都不是孤立的——缓存的语义相似度计算需要一次 embedding 调用，这个调用本身也需要走路由和重试。层与层之间有自己的依赖关系，设计时要整体看。

## 现在的理解

生产级 LLM 调用本质是在「成本、速度、质量、稳定性」四维空间里做架构权衡。没有万能配置，只有场景适配。
