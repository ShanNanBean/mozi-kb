# RAG 架构全景

## 最初的问题

我想搞清楚 RAG（检索增强生成）到底是怎么工作的——为什么市面上几乎所有知识库产品都是同一套「切片 → 向量化 → 检索 → 生成」的模式？

## 对话展开

### RAG 是什么

小墨说，RAG 的思路很直接：别让模型靠记忆回答，让它先去查资料，再拿着资料回答。类似开卷考试——给你一本参考书，边查边答。

大模型有两大硬伤：**知识截止日期**（训练数据是旧的）和**幻觉**（编造不存在的事实）。RAG 解决的就是这两个问题。

### 为什么分两段：离线 vs 在线

```
离线阶段（提前做）：
  文档 → 切片 → 向量化 → 存入向量数据库

在线阶段（用户提问时）：
  问题 → 向量化 → 检索Top-K相关片段 → 拼进Prompt → LLM生成回答
```

关键原因：不能把整本书塞进 prompt——一本书 10 万字，GPT-4 上下文才 128K token，而且 token 越贵越慢。所以必须「只塞相关的」。

### 切片策略比模型重要

这是我原先的误解——以为核心是选哪个 embedding 模型。小墨纠正：

| 切片策略 | 做法 | 适合 |
|----------|------|------|
| 固定长度 | 每 500 字切一刀 | 通用文章 |
| 按段落 | 以空行为界 | 结构化文档 |
| 语义切片 | 用模型判断语义边界 | 高质量场景 |
| 父子切片 | 检索小片段，回答时带上上下文 | 需要完整语义 |

换更好的 embedding 模型提升 5%，切片策略调好提升 30%。

### 检索策略也不是只有向量

简单的做法的确就是「问题转向量 → 找最相似的 K 个片段」。但用户说「上个月的更新」时，向量相似度找不到「时间」信息。

这时需要**混合检索**：向量相似度 + 关键词匹配（BM25），两者加权排序。

### 怎么喂给 LLM 也有讲究

不是把片段直接丢进去就行，需要组织成：
```
基于以下资料回答问题：
[片段1 来源：xxx]
...
问题：xxx
要求：如果资料不包含答案，就说不知道。
```

这句「不知道」很关键——没有它，模型会硬编。

### 为什么所有产品都用这套

三个现实约束：全量塞不进上下文（成本爆炸）、向量检索是性价比最高的、用户看不出差别。所以 90% 的知识库产品底层都一样，差异只在每个环节的优化深度。

## 关键转折

理解了切片本质上是**信息损耗换成本**——切开的过程中丢掉了文档结构、跨片段逻辑、长程依赖。这些在简单问答里不明显，但在需要全局理解的题目上（「对比第三章和第七章的观点」）会暴露问题。

## 现在的理解

- RAG 管道：切片 → 向量化 → 检索 → 拼 prompt → 生成
- 切片是核心瓶颈，不是模型
- 混合检索 > 纯向量检索
- 未来方向是知识图谱增强（Graph RAG）和长上下文模型对切片范式的冲击
