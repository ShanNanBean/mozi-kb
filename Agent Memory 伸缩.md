# Agent Memory 伸缩

## 最初的问题

随着时间变长，Agent Memory 越积越多，每次加载都会烧很多 token，而且考验模型上下文长度。怎么解决？

## 对话展开

### 量化问题

每天一条纠偏经验（200 字 ≈ 300 token），一年 365 × 300 = 109,500 token。GPT-4 128K 勉强装下但留给对话的不到 2 万 token。不解决死路一条。

### 四层解法

**第一层：检索式注入**

```
全量 Memory（1000条）
  → 用户当前问题向量化
  → Memory 向量检索 → Top 5 最相关
  → 只注入这 5 条（~1500 token）
```

风险：跨领域经验可能语义不够近，检索不到。

**第二层：时间降权**

```
最近 7 天 → 完整内容（200 token）
7-30 天   → 压缩摘要（50 token）
30+ 天    → 一句话（20 token）
```

365 条降权后：9250 token，比 10 万少了 90%。

**第三层：合并去重——相似经验归拢**

```
原始：
  - 「数组越界时检查栈大小」
  - 「栈溢出伪装成索引错误」
  - 「递归太深别死磕边界检查」
合并：
  - 「调试原则：3次同类失败→怀疑根因误判」
```

一条替代三条，信息密度更高。

**第四层：技能化——经验升级为 Skill**

```
Memory：「3次同类错误→根因重审」
  ↓ 用得多了
Skill：「debug-root-cause-review」
  ↓ Memory 删除，Skill 保留
```

这是最重要升级路径——把高频经验从被动注入变成按需加载。

### 四层组合效果

1000 条记忆 → 注入成本从 30 万 token 降到约 3000-5000 token。

### Obsidian 笔记不会爆

因为笔记不走全量注入——只在搜索、点链接、session_search 时才触发检索。真正会爆的是 Memory 工具存的内容（每次对话都注入）。

所以 Memory 只存稳定事实：
```
✓ 「用户偏好实战优先」
✗ 「2026-05-20 学了 RAG 架构」→ 放 Obsidian
```

## 关键转折

整个系统——缓存降权、根因重审、Memory 伸缩——都指向同一个模式：**需要感知「什么时候该怀疑自己」的机制。** 这不是 LLM 的事，是 Agent 架构的事。

## 现在的理解

Agent Memory 不是 memory 的问题，是信息管理的问题。检索、降权、合并、升级——每一层都在做同一件事：用最小的 token 代价保留最大的纠偏价值。
